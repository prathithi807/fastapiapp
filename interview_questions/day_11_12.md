# 🐳 Docker & AWS Deployment (Day 11–12) Interview Questions

This question bank contains a complete set of possible interview questions covering Dockerization, multi-stage builds, orchestration with Docker Compose, AWS Elastic Beanstalk deployments, secrets management, and persistent storage using AWS S3.

---

## 📌 Section 1: Docker & Multi-Container Workflows

### Q1. What is the key difference between a Virtual Machine (VM) and a Docker Container?
* **Virtual Machine (VM)**:
  * Contains a full copy of a guest operating system, virtual hardware drivers, and the application.
  * Runs on top of a hypervisor.
  * Consumes gigabytes of disk space and takes minutes to boot.
  * High resource overhead.
* **Docker Container**:
  * Shares the host operating system kernel directly with other containers.
  * Runs as an isolated process in userspace.
  * Consumes megabytes of space and boots in milliseconds.
  * Low resource overhead, high density.

---

### Q2. What are Multi-Stage Builds, and why are they crucial for React/Vite frontend containerization?
* **Multi-Stage Builds**: A technique where you use multiple `FROM` instructions in a single `Dockerfile`. Each `FROM` starts a new stage of the build using a different base image. You can selectively copy artifacts (like compiled files) from one stage to another, leaving behind temporary compilers and source files.
* **Crucial for React**:
  * **Stage 1 (Build Stage)**: Uses a heavy Node.js image to download dependencies (`node_modules`) and compile TypeScript/Vite source code into static html/css/js bundles.
  * **Stage 2 (Runtime Stage)**: Uses a lightweight web server image (like `nginx:alpine`). It only copies the compiled static build folder from Stage 1.
  * **Result**: The final production image contains only Nginx and static files (approx. 20-30MB), keeping the deployment fast, secure, and devoid of Node.js development dependencies.

---

### Q3. Explain the difference between `EXPOSE` and port mapping (`-p` or `ports:`) in Docker Compose.
* **`EXPOSE`**: An informational instruction in a `Dockerfile` that documents which port the application inside the container listens on at runtime. It does not actually open or map any ports on the host machine.
* **Port Mapping (`-p host_port:container_port` or `ports:`)**: Instructs the Docker host engine to open a specific port on the host machine’s network card and forward all incoming traffic to the designated port inside the running container.

---

### Q4. What is the role of Docker Volumes, and what is the difference between Named Volumes and Bind Mounts?
* **Volumes**: Mechanisms used to persist data generated by and used by Docker containers outside the container’s writeable layer.
* **Named Volumes**:
  * Managed entirely by Docker on the host disk (typically under `/var/lib/docker/volumes/`).
  * Recommended for database storage (like PostgreSQL data folders) because they are isolated from host filesystem quirks.
* **Bind Mounts**:
  * Maps an exact user-specified folder on the host machine to a folder inside the container (e.g. `./backend:/app`).
  * Recommended for local active development because code modifications on the host are immediately reflected inside the running container (hot reloading).

## ☁️ Section 2: AWS Elastic Beanstalk & Cloud Deployment

### Q5. What is AWS Elastic Beanstalk, and how does it simplify deploying containerized applications?
* **AWS Elastic Beanstalk**: A Platform-as-a-Service (PaaS) that automatically handles deployment details like provisioning virtual machines (EC2), load balancing, auto-scaling, SSL certificates, application health monitoring, and logging.
* **Simplification**: Instead of configuring AWS ECS, load balancers, and Auto-Scaling groups manually, developers can upload a `docker-compose.yml` or a `Dockerrun.aws.json` file. Beanstalk parses it, sets up the instances, and exposes a live public URL.

---

### Q6. How do you handle database schema migrations (e.g. Alembic upgrade head) during automated Elastic Beanstalk deployments?
* **The Challenge**: If you have 3 instances running in an auto-scaling group and all boot at once, they might run migrations concurrently, causing database locking or migration corruption.
* **Solutions**:
  1. **Leader Action (Container Command)**: In Elastic Beanstalk configurations (like `.ebextensions/`), designate one instance as the "leader" and configure it to run `alembic upgrade head` before booting the rest of the application.
  2. **Init Containers / Entrypoint Scripts**: Run a pre-migration container step that executes the migrations, locking the database schema before the main application containers start.
  3. **CI/CD Pipeline**: Run migrations from your deployment pipeline (e.g., GitHub Actions) directly against the database before pushing the new code deployment to Beanstalk.

## 🔒 Section 3: Secrets Management & AWS S3 Storage

### Q7. How should database passwords and API keys be injected into production containers on AWS? Why is checking them into a `Dockerfile` a critical security failure?
* **Why not check into Dockerfile**: Docker images are stored in registries (like Docker Hub or AWS ECR). Anyone with access to the registry can pull the image and use `docker inspect` to read the environment variables or view the layers to steal the secrets.
* **Best Practices**:
  1. **Elastic Beanstalk Environment Properties**: Inject secrets as environment variables using the AWS Beanstalk console. These are securely passed to the container at launch.
  2. **AWS Systems Manager (SSM) Parameter Store / AWS Secrets Manager**: The container fetches secrets programmatically at startup using IAM authorization.

---

### Q8. What is AWS S3? How do containerized backends authenticate with S3 securely on AWS without hardcoding keys?
* **AWS S3 (Simple Storage Service)**: An object storage service built to store and retrieve any amount of data (e.g., user resumes, images, files). It is crucial because container filesystems are ephemeral.
* **Secure Authentication (IAM Roles)**:
  * Create an **IAM Role** with policies allowing read/write actions on the specific S3 bucket.
  * Attach this role to the EC2 instances running your containers (as an **Instance Profile**).
  * In the FastAPI application, use the Python library `boto3`. If you do not specify credentials, `boto3` automatically looks up the instance profile metadata credentials at runtime, letting the container authenticate securely without hardcoding any keys inside the repository.

## 📐 Section 4: Code & Scenario Questions

### Q9. Write a multi-stage `Dockerfile` for a React Vite production build served using Nginx.
* **Answer**:
```dockerfile
# Stage 1: Build the React Application
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: Serve the Static Files using Nginx
FROM nginx:stable-alpine
COPY --from=builder /app/dist /usr/share/nginx/html
# Copy custom nginx configuration to handle React Router routing
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

### Q10. Write a `docker-compose.yml` file that orchestrates:
1. A **PostgreSQL database** (exposing port 5433 host, 5432 container) with a healthcheck.
2. A **FastAPI backend** that depends on the database health check.
* **Answer**:
```yaml
version: '3.8'

services:
  db:
    image: postgres:15-alpine
    environment:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: password
      POSTGRES_DB: student_db
    ports:
      - "5433:5432"
    volumes:
      - postgres_data:/var/pn/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres -d student_db"]
      interval: 5s
      timeout: 5s
      retries: 5

  backend:
    build: ./backend
    ports:
      - "8000:8000"
    environment:
      - DATABASE_URL=postgresql+asyncpg://postgres:password@db:5432/student_db
    depends_on:
      db:
        condition: service_healthy

volumes:
  postgres_data:
```

### Q11. Scenario: During local docker-compose spin-up, your FastAPI container crashes instantly with `ConnectionRefusedError: [Errno 111] Connect call failed` pointing to the database port. Your PostgreSQL container is up and running. What is causing this, and how do you resolve it?
* **Why it happens**: A race condition. The PostgreSQL container starts up and opens its port 5432. The FastAPI container sees the port is open and starts booting. However, PostgreSQL needs an extra 3–5 seconds to perform internal initialization (loading logs, starting processes, creating tables). While Postgres is initializing, it rejects incoming TCP connections. Since FastAPI tries to connect instantly at boot, it crashes.
* **How to Resolve**:
  1. **Add Database Healthcheck**: In `docker-compose.yml`, define a `healthcheck` on the database service using `pg_isready`, and use `condition: service_healthy` in the backend's `depends_on` (as shown in the Q10 example).
  2. **Use Wait Scripts**: Wrap the backend’s startup command in a script like `wait-for-it.sh` or a custom python ping loop that polls the database connection until it succeeds before running the server command (`uvicorn`).

* **Answer**:

     * Port Mapping: `"5433:5432"` (binds host port 5433 to PostgreSQL's native port 5432 inside the container network to avoid conflicts with any local host Postgres instances).

     * Port Mapping: `"3000:80"` (maps host port 3000 to the container's Nginx port 80).

### Q13. Why does the `backend` service in our `docker-compose.yml` declare `host.docker.internal:host-gateway` under `extra_hosts`?
* **Answer**:
  * **The Purpose**: To allow the FastAPI container to communicate with a local LLM service (like Ollama) running directly on the developer's host machine (outside the Docker bridge network).
  * **How it works**: Containers in a default bridge network cannot access the host machine via `localhost` (since `localhost` resolves to the container itself). By adding `host.docker.internal:host-gateway` to `extra_hosts`, Docker adds a mapping to the container's `/etc/hosts` file resolving `host.docker.internal` to the host machine's IP address. This enables the backend to route queries to `OLLAMA_HOST=host.docker.internal:11434`.

---

### Q14. In our compose configuration, how does the frontend container learn the backend API URL during image assembly?
* **Answer**: It is passed as a build argument in `docker-compose.yml`:
```yaml
frontend:
  build:
    context: ./frontend/talentspark
    dockerfile: Dockerfile
    args:
      - VITE_API_URL=http://localhost:8000
```
Vite compiles environment variables at build-time (rather than runtime). When Docker compiles the React frontend inside the multi-stage build, the arg `VITE_API_URL` is fetched by the `Dockerfile` and burned directly into the compiled JavaScript production assets served by Nginx.

---

### Q15. Where is PostgreSQL database data stored in our compose setup?
* **Answer**: It is persisted using a named volume called `postgres_data`:
```yaml
services:
  db:
    ...
    volumes:
      - postgres_data:/var/lib/postgresql/data
```
This maps the Docker-managed volume `postgres_data` directly to `/var/lib/postgresql/data` (which is the default directory where PostgreSQL writes database files). This guarantees that user profiles, companies, and jobs are not lost when containers are stopped or removed.
