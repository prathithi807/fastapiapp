## 📌 Section 1: RAG & Text Embeddings

### Q1. What is Retrieval-Augmented Generation (RAG), and why is it useful?
* **RAG**: An architectural pattern that optimizes the output of a Large Language Model by querying an authoritative, external knowledge source (like a database or document store) before generating the response.
* **Why it is useful**:
  * **Eliminates Hallucinations**: Grounds the LLM's responses in factual, retrieved documents.
  * **Solves Context Window Limits**: Instead of feeding thousands of documents into the prompt (which is expensive and exceeds limits), RAG retrieves only the top 3-5 most relevant text chunks.
  * **Private Data Integration**: Allows querying internal company documents or user-specific data that the public LLM was never trained on.
  * **Real-time Data Updates**: You can add, edit, or delete documents in the knowledge base instantly without needing to retrain or fine-tune the LLM.

---

### Q2. What are text embeddings? How do they represent semantic meaning?
* **Text Embeddings**: Numerical representations of text (sentences, paragraphs, or entire documents) structured as high-dimensional vectors (arrays of floating-point numbers, e.g., 384 or 1536 dimensions).
* **Semantic Meaning**: Deep learning models (like BERT or OpenAI's text-embedding models) place semantically similar words or sentences close to each other in this high-dimensional vector space.
  * *Example*: The vector representation of `"software developer"` will be mathematically closer to `"programmer"` than to `"banana"`.

---

### Q3. Explain semantic search vs. keyword search. When should you use each?
* **Keyword (Lexical) Search (e.g., SQL LIKE, BM25)**:
  * Looks for exact character matches between the query and the document.
  * *When to use*: Precise searches (e.g., searching for a specific serial number, user ID, or exact legal term).
  * *Limitation*: Fails if synonyms are used (e.g., searching for "job" won't find descriptions that only mention "career" or "vacancy").
* **Semantic Search**:
  * Looks for meaning and intent regardless of the exact words used.
  * *When to use*: Natural language queries (e.g., searching for "remote coding jobs" returns listings containing "work from home software engineer").
  * *Limitation*: Can be computationally slower and might miss highly specific alphanumeric identifiers.

## ⚙️ Section 2: Vector Databases & Retrieval Pipelines

### Q4. What is a Vector Database (e.g., Qdrant, ChromaDB)? How does it differ from a Relational Database (PostgreSQL)?
* **Vector Database**: Optimized specifically for storing and querying multi-dimensional vector embeddings. It uses specialized index algorithms (like HNSW - Hierarchical Navigable Small World) to perform rapid **Approximate Nearest Neighbor (ANN)** searches across millions of vectors in milliseconds.
* **Relational Database**: Optimized for tabular structured data, transactions, and exact indexing keys (B-Trees). Doing a vector distance search across millions of rows in a traditional SQL database is extremely slow because it requires calculating distance row-by-row unless specialized extensions (like `pgvector`) are installed.

---

### Q5. What distance metrics are used to compare vectors? Explain Cosine Similarity.
* **Common Metrics**:
  * **Cosine Similarity**: Measures the cosine of the angle between two vectors. It measures orientation rather than magnitude. The score ranges from `-1` to `1` (or `0` to `1` in normalized spaces), where `1` represents identical direction.
  * **Dot Product**: Multiplies corresponding coordinates and sums them. Faster to compute but depends on vector magnitude.
  * **Euclidean Distance (L2)**: Measures the straight-line distance between two points in space. Smaller distance means higher similarity.
* **Cosine Similarity Usage**: Preferred for text classification and RAG because it evaluates similarity based on topic/content, ignoring document length differences (which alter vector magnitudes).

---

### Q6. What is FastEmbed, and what are its advantages for local development compared to cloud APIs?
* **FastEmbed**: A lightweight Python library developed by Qdrant for generating text embeddings locally.
* **Advantages**:
  * **Zero Cost**: Does not charge per token like OpenAI's embeddings API.
  * **Offline Execution**: Runs locally without making network calls, resolving security/privacy concerns.
  * **Performance**: Built on ONNX runtime, meaning it runs extremely fast on standard CPUs without requiring heavy machine learning frameworks like PyTorch or TensorFlow.

## 🤝 Section 3: Semantic Matching & AI Pipeline Integration

### Q7. Explain the step-by-step pipeline of a RAG query inside a backend application.
* **Answer**:
  1. **User Query**: The user asks a question (e.g., `"Which jobs match my Python background?"`).
  2. **Embedding**: The query string is passed to an embedding model (e.g., FastEmbed) to generate a query vector.
  3. **Vector Retrieval**: The query vector is sent to the Vector Database (e.g., Qdrant). The database returns the top $K$ (e.g., top 5) documents whose vectors have the highest cosine similarity.
  4. **Context Construction**: The backend extracts the text payload from the retrieved vector metadata and formats it into a prompt string.
  5. **LLM Synthesis**: The prompt (System instructions + Context + User query) is sent to the LLM (e.g., Groq / OpenAI).
  6. **Response**: The LLM synthesizes a fluent, natural language answer based *only* on the retrieved context, returning it to the user.

---

### Q8. How do you implement a smart semantic job matching feature?
* **Answer**:
  1. Create a Qdrant collection containing all job listings, storing the job attributes in the vector payload.
  2. Have the candidate input their skills and experience.
  3. Concatenate these inputs into a profile string (e.g. `"Skills: Python, SQL. Experience: 3 years building APIs."`).
  4. Embed this profile string into a vector.
  5. Query the Qdrant collection using the profile vector.
  6. Return the matching jobs ranked by similarity score.

### Q9. Write a FastAPI endpoint that takes a search query, embeds it locally, searches a Qdrant collection, and formats the results for a React UI.
* **Answer**:
```python
from fastapi import APIRouter, HTTPException
from pydantic import BaseModel
from qdrant_client import QdrantClient
from fastembed import TextEmbedding

router = APIRouter(prefix="/search", tags=["Search"])
qdrant_client = QdrantClient(url="http://localhost:6333")
embedding_model = TextEmbedding("BAAI/bge-small-en-v1.5")

class SearchRequest(BaseModel):
    query: str
    top_k: int = 5

@router.post("/")
async def semantic_search(request: SearchRequest):
    try:
        # 1. Embed user query locally (returns list of vectors)
        query_vector = next(embedding_model.embed([request.query])).tolist()
        
        # 2. Query Qdrant
        hits = qdrant_client.query_points(
            collection_name="job_descriptions",
            query=query_vector,
            limit=request.top_k
        )
        
        # 3. Format results for frontend
        results = []
        for hit in hits.points:
            results.append({
                "job_id": hit.payload.get("job_id"),
                "title": hit.payload.get("title"),
                "description": hit.payload.get("description"),
                "score": round(hit.score, 4)
            })
        return {"results": results}
    except Exception as e:
        raise HTTPException(status_code=500, detail=f"Search failed: {str(e)}")
```

### Q10. Write the React TypeScript code for a Resume Analyzer page that calls a backend endpoint `/rag/analyse-resume` and displays the markdown analysis.
* **Answer**:
```typescript
import React, { useState } from "react";
import axios from "axios";

interface AnalysisResponse {
    analysis: string; // Returns markdown string containing skills, suggestions
}

export const ResumeAnalyzer = () => {
    const [resumeText, setResumeText] = useState<string>("");
    const [analysisResult, setAnalysisResult] = useState<string>("");
    const [loading, setLoading] = useState<boolean>(false);
    const [error, setError] = useState<string | null>(null);

    const handleAnalyze = async () => {
        setLoading(true);
        setError(null);
        setAnalysisResult("");
        try {
            const token = localStorage.getItem("access_token");
            const response = await axios.post<AnalysisResponse>(
                "http://localhost:8000/rag/analyse-resume",
                { resume_text: resumeText },
                { headers: { Authorization: `Bearer ${token}` } }
            );
            setAnalysisResult(response.data.analysis);
        } catch (err: any) {
            setError(err.response?.data?.detail || "Failed to analyze resume.");
        } finally {
            setLoading(false);
        }
    };

    return (
        <div style={{ maxWidth: "800px", margin: "0 auto", padding: "20px" }}>
            <h2>AI Resume Analyzer</h2>
            <textarea
                value={resumeText}
                onChange={(e) => setResumeText(e.target.value)}
                placeholder="Paste your resume plain text here..."
                rows={12}
                style={{ width: "100%", padding: "10px", marginBottom: "15px" }}
            />
            <button onClick={handleAnalyze} disabled={loading || !resumeText.trim()}>
                {loading ? "Analyzing..." : "Analyze Resume"}
            </button>

            {error && <p style={{ color: "red" }}>{error}</p>}

            {analysisResult && (
                <div style={{ marginTop: "30px", border: "1px solid #ddd", padding: "20px", background: "#f9f9f9" }}>
                    <h3>Analysis Result</h3>
                    {/* Render raw text preserving line breaks, or integrate a markdown renderer */}
                    <div style={{ whiteSpace: "pre-wrap" }}>{analysisResult}</div>
                </div>
            )}
        </div>
    );
};
```

### Q11. Scenario: Your RAG pipeline returns irrelevant search results that do not answer the user's question. How do you debug and optimize the retrieval step?
* **Answer**:
  1. **Chunking Strategy**: Check chunk sizes. If chunks are too small, they lack surrounding context. If they are too large, the semantic signal gets diluted. Adjust chunk overlap to ensure text at boundary lines is not lost.
  2. **Embedding Model Quality**: Evaluate if the embedding model matches the domain (e.g. using a model fine-tuned on code, medical terminology, or legal documents).
  3. **Query Expansion / Query Rewriting**: Users often write poorly formatted queries. Use an LLM to rewrite the user query (e.g., adding synonyms or translating to technical terms) before embedding it.
  4. **Reranking (Cross-Encoders)**: Use a lightweight Bi-Encoder (like BGE) to retrieve the top 20 candidates quickly, then apply a computationally heavy Cross-Encoder (Reranker) to rank those 20 candidates precisely.
  5. **Metadata Filtering**: Apply pre-filtering rules (such as matching regions, languages, or roles) in Qdrant before conducting the vector similarity search to narrow the search scope.

            info = qdrant.get_collection(COLLECTION_NAME)
            existing_size = info.config.params.vectors.size
            if existing_size != VECTOR_SIZE:

### Q13. Walk through the job indexing logic in `embed_all_jobs`. How are database records converted into vector points?
* **Answer**: In `embed_all_jobs`:
  1. It reads all jobs from the PostgreSQL database using an async query: `await db.execute(select(Job))`.
  2. It iterates through the jobs, constructing a combined search string: `text = f"{job.title} {job.description or ''}"`.
  3. It calls `embed_text` to generate a 384-dimensional floating point list.
  4. It constructs a `PointStruct` where the vector ID matches the SQL database `job.id`, and appends the relevant text details to the payload:
     ```python
     PointStruct(
         id=job.id,
         vector=vector,
         payload={"title": job.title, "description": job.description or "", "salary": job.salary, "job_id": job.id}
     )
     ```
  5. It bulk upserts the list into Qdrant: `qdrant.upsert(collection_name=COLLECTION_NAME, points=points)`.
