## 📌 Section 1: LLM Fundamentals & Prompt Design

### Q1. What are tokens, context windows, and how do they affect LLM application development?
* **Tokens**: The basic building blocks used by LLMs to process text. A token is not necessarily a full word; it can be a sub-word, a single character, or a punctuation mark (roughly 1 token = 4 characters or 0.75 words in English).
  * *Impact*: LLM providers bill applications based on the number of input and output tokens.
* **Context Window**: The maximum number of tokens a model can process in a single request-response cycle (includes both the user query, system instructions, chat history, and the generated output).
  * *Impact*: If the prompt history exceeds the context window, the model will cut off older parts of the conversation (context truncation) or fail with an out-of-context error. Developers must manage history pruning or use RAG (Retrieval-Augmented Generation) to stay within limits.

---

### Q2. What is the `temperature` parameter in LLMs? When would you use a value of `0` vs `1`?
* **Temperature**: A hyperparameter that controls the randomness or creativity of the model's generated responses (ranging between `0` and `1` or `2`).
  * **Temperature = 0**: The model becomes deterministic, always choosing the token with the highest mathematical probability.
    * *Use Case*: Code generation, database query writing, structured JSON outputs, factual search.
  * **Temperature = 1 (or higher)**: The model introduces random variance, leading to more creative, diverse, and unpredictable responses.
    * *Use Case*: Creative writing, roleplay chatbots, brainstorming, naming suggestions.

---

### Q3. Explain the roles of System, User, and Assistant messages in a chat completion prompt.
* **System Prompt**: Sets the behavior, persona, rules, and boundaries of the AI assistant (e.g., `"You are a bilingual database assistant. Only return answers in SQL formats"`). It cannot be easily overridden by the user.
* **User Message**: The direct question, prompt, or command input by the end user (e.g., `"Find all jobs paying over 80k"`).
* **Assistant Message**: The previous response generated by the LLM. Storing assistant messages is crucial for maintaining conversation history in multi-turn dialogues.

## ⚙️ Section 2: OpenAI API & Streaming Responses

### Q4. What is Server-Sent Events (SSE), and how does LLM streaming work under the hood?
* **Server-Sent Events (SSE)**: A lightweight HTTP standard that allows a web server to push real-time, one-way data streams to a client over a single long-lived HTTP connection.
* **How LLM Streaming Works**:
  1. Instead of waiting 10 seconds for the model to generate a full response, the server sets the header `Content-Type: text/event-stream`.
  2. The LLM processes tokens one-by-one and pushes them to the backend server.
  3. The backend server immediately flushes these tokens to the client browser formatted as SSE chunks (e.g., `data: {"text": "hello"}\n\n`).
  4. The client's browser appends the chunks onto the screen in real-time, giving the user a "typing" experience and lowering the time-to-first-token (TTFT).

---

### Q5. What is LLM Function Calling (or Tool Calling)?
* **Answer**: Function calling is a feature where you describe native programming functions (their name, parameters, and JSON schemas) to the LLM.
* **How it works**:
  1. The user asks a question requiring external data (e.g., `"What is the salary for Job ID 5?"`).
  2. Instead of answering directly, the LLM recognizes that a function fits the query, halts execution, and returns a JSON payload containing the function name and arguments (e.g., `{"name": "get_job_salary", "arguments": {"job_id": 5}}`).
  3. Your backend executes the local function using those arguments, queries the database, and gets the result (e.g., `$95,000`).
  4. The backend sends the result back to the LLM.
  5. The LLM uses that result to formulate a natural language answer (e.g., `"Job ID 5 pays $95,000 per year."`).
* *Note: The LLM does not run the code; it only generates the arguments schema.*

## 🦜 Section 3: LangChain Architecture & Components

### Q6. What is LCEL (LangChain Expression Language)? How does the pipe operator `|` work?
* **LCEL**: A declarative syntax in LangChain used to construct complex chains of LLM tasks.
* **Pipe Operator (`|`)**: Overloads Python’s bitwise OR operator to chain components together. The output of the left component is automatically passed as the input of the right component.
```python
# A simple LCEL chain:
chain = prompt | llm | output_parser
```
*Behind the scenes*: LangChain implements the Runnable protocol. Each component inherits from `Runnable`, which defines input/output types and standard methods like `.invoke()`, `.stream()`, and `.batch()`.

---

### Q7. How does LangChain manage conversation history (Memory)?
* **Answer**: LangChain stores previous messages in a database or in-memory array and automatically formats them back into the prompt during subsequent calls.
* **Memory Strategies**:
  * **ConversationBufferMemory**: Keeps the entire history of messages. Simple, but quickly consumes the context window.
  * **ConversationSummaryMemory**: Uses a secondary LLM call to periodically write a condensed summary of the conversation, saving context tokens.
  * **RunnableWithMessageHistory**: The modern (v0.2+) LangChain way to dynamically fetch and store session history using external databases (like Redis or PostgreSQL).

---

### Q8. What are Output Parsers in LangChain?
* **Answer**: Structured output parsers instruct the LLM to format its response according to a specific schema (such as JSON or CSV) and parse that output string back into a Python object or Pydantic model.
* **Use Case**: Extracting data from unstructured text (e.g., parsing a resume text to extract skills and experience).

## 📐 Section 4: Code & Scenario Questions

### Q9. Implement an asynchronous FastAPI endpoint that streams responses from OpenAI using Server-Sent Events (SSE).
* **Answer**:
```python
import os
from fastapi import APIRouter
from fastapi.responses import StreamingResponse
from openai import AsyncOpenAI

router = APIRouter()
openai_client = AsyncOpenAI(api_key=os.getenv("OPENAI_API_KEY"))

async def openai_stream_generator(prompt: str):
    response = await openai_client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
        stream=True  # Enables chunk-by-chunk streaming
    )
    
    async for chunk in response:
        content = chunk.choices[0].delta.content
        if content is not None:
            # SSE protocol format: prefix with 'data: ' and end with double newline
            yield f"data: {content}\n\n"

@router.get("/chat/stream")
async def stream_chat(prompt: str):
    return StreamingResponse(
        openai_stream_generator(prompt),
        media_type="text/event-stream"
    )
```

### Q10. Write the corresponding React component that reads the SSE stream using the `fetch` API and renders it chunk-by-chunk with a typing effect.
* **Answer**:
```typescript
import React, { useState } from "react";

export const StreamingChat = () => {
    const [prompt, setPrompt] = useState("");
    const [response, setResponse] = useState("");
    const [loading, setLoading] = useState(false);

    const handleSend = async () => {
        setResponse("");
        setLoading(true);
        try {
            const apiResponse = await fetch(`http://localhost:8000/chat/stream?prompt=${encodeURIComponent(prompt)}`);
            
            if (!apiResponse.body) return;
            const reader = apiResponse.body.getReader();
            const decoder = new TextDecoder();
            let done = false;

            while (!done) {
                const { value, done: doneReading } = await reader.read();
                done = doneReading;
                const chunkValue = decoder.decode(value);
                
                // Parse the SSE formatted lines (data: <text>\n\n)
                const lines = chunkValue.split("\n\n");
                lines.forEach((line) => {
                    if (line.startsWith("data: ")) {
                        const text = line.replace("data: ", "");
                        setResponse((prev) => prev + text);
                    }
                });
            }
        } catch (err) {
            console.error("Streaming failed", err);
        } finally {
            setLoading(false);
        }
    };

    return (
        <div>
            <textarea value={prompt} onChange={(e) => setPrompt(e.target.value)} />
            <button onClick={handleSend} disabled={loading}>
                {loading ? "Thinking..." : "Send"}
            </button>
            <div className="response-box" style={{ whiteSpace: "pre-wrap", border: "1px solid #ddd", padding: "15px" }}>
                {response}
            </div>
        </div>
    );
};
```

### Q11. Scenario: You are deploying an LLM app. The API call costs are skyrocketing. How do you limit or optimize costs at the application architecture level?
* **Answer**:
  1. **Semantic Caching (Redis)**: Save prompts and their generated responses in a Redis Vector Database. Before calling OpenAI, run a similarity search on the prompt. If a user asks a similar question (e.g. `"How to write a resume"` vs `"How do I write a resume"`), return the cached response for free without hitting OpenAI.
  2. **Model Tiering**: Route complex, reasoning-heavy tasks (like code generation) to expensive models (like GPT-4o), while routing simple classifier/routing questions to cheaper models (like GPT-4o-mini).
  3. **Context Pruning**: Implement token limits on the conversation history. Remove intermediate messages or summarize them before feeding them to the next API call to avoid input-token inflation.
  4. **Strict Token Limits**: Set `max_tokens` constraints on completions to prevent long, runaway outputs.

### Q13. How is session-based message history implemented in `langchai_service.py`? Where are messages stored?
* **Answer**:
  * **Memory Storage**: Messages are stored in-memory using a dictionary called `store`:
    ```python
    store = {}
    ```
  * **History Retrieval**: The `get_history` helper checks if a session ID exists in the store. If not, it creates a new `ChatMessageHistory` instance:
    ```python
    def get_history(session_id: str) -> ChatMessageHistory:
        if session_id not in store:
            store[session_id] = ChatMessageHistory()
        return store[session_id]
    ```
  * **Runnable Connection**: It wraps the chain using LangChain's `RunnableWithMessageHistory` to bind the session history retrieval to the execution execution loop:
    ```python
    chat_with_memory = RunnableWithMessageHistory(
        runnable=chain_with_memory,
        get_session_history=get_history,
        input_messages_key="user_query",
        history_messages_key="chat_history"
    )
    ```

### Q14. Explain the LCEL chain composition in our chat service.
* **Answer**:
  1. We define the input template using `ChatPromptTemplate.from_messages`:
     ```python
     prompt_with_memory = ChatPromptTemplate.from_messages([
         ("system", "You are a helpful career guidance assistant."),
         ("placeholder", "{chat_history}"),
         ("human", "{user_query}")
     ])
     ```
     This structure contains the system persona, standard placeholder for history injection, and the latest human query.
  2. We build the chain by piping the prompt directly to the LLM instance:
     ```python
     chain_with_memory = prompt_with_memory | llm
     ```
  3. When `ask_career_chatbot_response(question, session_id)` is invoked, it triggers `.invoke` passing user input and config:
     ```python
     response = chat_with_memory.invoke(
         {"user_query": question},
         {"configurable": {"session_id": session_id}}
     )
     ```
