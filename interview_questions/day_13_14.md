## 📌 Section 1: Capstone Walkthrough & Pitching

### Q1. "Tell me about TalentSpark—what problem does it solve and what is its value proposition?" (The 2-Minute Pitch)
* **The Pitch**:
  * **The Problem**: Job seeking and recruitment are highly inefficient. Candidates spend hours tailoring resumes and scrolling through keyword-based job boards that miss actual matches. Recruiters get flooded with thousands of mismatched resumes, causing severe hiring latency.
  * **The Solution (TalentSpark)**: TalentSpark is a full-stack, AI-powered talent matching platform. It bridges the gap by replacing keyword searches with **semantic vector searches** (meaning-based job and profile matching).
  * **Core Features**:
    1. **AI Resume Analyzer**: Processes unstructured resumes, highlighting key skills, strengths, and areas to improve.
    2. **Smart Job Match**: Converts candidate profiles into vector embeddings and matches them with jobs based on cosine similarity, showing a percentage score.
    3. **Career Q&A Bot**: An AI career counselor utilizing LangChain with session-based memory to answer job hunting and application questions.
    4. **Enterprise Role Management**: Secure, role-based CRUD operations where recruiters post/manage jobs and candidates search/apply.
  * **The Tech Stack**: Built using FastAPI, PostgreSQL, SQLAlchemy Async ORM, React (Vite + TypeScript), Qdrant Vector DB, LangChain, Groq/OpenAI APIs, and Dockerized for AWS Elastic Beanstalk.

---

### Q2. Walk me through the architecture of TalentSpark. How does data flow from the client to the database and AI services?
* **Answer**:
  1. **Frontend (Client)**: A React (Vite + TS) SPA served via Nginx. It communicates with the backend via Axios, storing the JWT auth token in secure storage.
  2. **API Layer (Reverse Proxy / Load Balancer)**: Directs client traffic.
  3. **Backend App (FastAPI)**: Runs asynchronously on Uvicorn. It handles routing, authorization dependencies, and requests validation via Pydantic.
  4. **Relational Database (PostgreSQL)**: Handles persistent transactional data (Users, Companies, Jobs) queried asynchronously via SQLAlchemy ORM.
  5. **Vector Database (Qdrant)**: Stores job descriptions and profile embeddings.
  6. **AI Orchestration Layer (LangChain / Local Embeddings)**: Computes vectors locally via FastEmbed and forwards LLM queries (Chatbot, Resume Analyzer) to external LLM providers (Groq/OpenAI).

## 🛡️ Section 2: API Hardening & Production Security

### Q3. How did you harden the TalentSpark API for production?
* **Answer**:
  * **Input Validation**: Strictly enforced Pydantic validation on all incoming request payloads, returning structured `422 Unprocessable Entity` errors for malformed requests.
  * **Robust Exception Handling**: Wrapped all router operations in custom `try-except` blocks, executing database rollbacks (`await db.rollback()`) on transaction failures to prevent database pool corruption and raising clean HTTP errors.
  * **Role-Based Access Control (RBAC)**: Implemented strict token extraction and role verification dependencies (`role_required(["admin", "recruiter"])`), preventing Candidate accounts from calling recruiter endpoints.
  * **Rate Limiting**: Configured rate-limiting parameters on AI endpoints to prevent Denial-of-Service (DoS) attacks and curb API bill inflation.

### Q4. How do you implement rate limiting on your AI endpoints in FastAPI?
* **Answer**: Use middleware or libraries like `slowapi` (which wraps `limits` in Python) to limit the number of requests a user can make within a time window (e.g., 5 requests per minute for LLM analysis).
```python
from slowapi import Limiter, _rate_limit_exceeded_handler
from slowapi.util import get_remote_address
from slowapi.errors import RateLimitExceeded

limiter = Limiter(key_func=get_remote_address)

# Inside main.py:
app.state.limiter = limiter
app.add_exception_handler(RateLimitExceeded, _rate_limit_exceeded_handler)

# Inside router:
@router.post("/analyse-resume")
@limiter.limit("5/minute")
def resume_analyse(request: ResumeRequest):
    ...
```
*Behind the scenes*: In production, slowapi uses a fast storage backend like **Redis** to keep track of request counts associated with client IP addresses or JWT user IDs.

## ⚛️ Section 3: React Production Polish & Error Boundaries

### Q5. What is a React Error Boundary? How does it prevent the entire application from crashing?
* **Error Boundary**: A React component that catches JavaScript errors anywhere in its child component tree, logs those errors, and displays a fallback UI instead of letting the entire application crash or render a blank white screen.
* **Mechanism**: It utilizes class component lifecycle methods `static getDerivedStateFromError()` (to render a fallback UI) and `componentDidCatch()` (to log error details).
```typescript
import React, { Component, ErrorInfo, ReactNode } from "react";

interface Props { children: ReactNode; }
interface State { hasError: boolean; }

export class ErrorBoundary extends Component<Props, State> {
  public state: State = { hasError: false };

  public static getDerivedStateFromError(_: Error): State {
    return { hasError: true };
  }

  public componentDidCatch(error: Error, errorInfo: ErrorInfo) {
    console.error("Uncaught error:", error, errorInfo);
  }

  public render() {
    if (this.state.hasError) {
      return <h1>Something went wrong. Please refresh the page.</h1>;
    }
    return this.children;
  }
}
```
*Note: In modern React apps, we wrap main layouts or individual widgets (like the chat bot card) inside `<ErrorBoundary>` components to isolate component crashes.*

## 📐 Section 4: System Design & Scaling to 100,000 Users

### Q6. "How would you scale TalentSpark to handle 100,000 active users?" (System Design Extension)
* **Answer**: You scale the application horizontally by addressing the bottlenecks at each tier of the stack:

#### 1. API & Compute Tier (FastAPI)
* Run multiple instances of the FastAPI container across virtual machines (e.g., AWS EC2 instances managed by an Auto-Scaling Group).
* Place an **Application Load Balancer (ALB)** in front of the instances to distribute client requests using Round Robin.
* Move long-running tasks (like PDF parsing or bulk embedding updates) out of the HTTP request-response cycle into an asynchronous worker queue (such as **Celery** or **ARQ** backed by **Redis**).

#### 2. Relational Database Tier (PostgreSQL)
* Implement **Read Replicas**: Direct write operations (`POST`, `PUT`, `DELETE`) to a primary writer instance, and route read queries (`GET`) to multiple read-only replica instances.
* Configure **Connection Pooling**: Use tools like **PgBouncer** in front of PostgreSQL to prevent running out of database connections as concurrent API workers scale up.
* Implement **Caching**: Use **Redis** to cache database responses for high-traffic, slow-changing queries (e.g., popular job listings).

#### 3. Vector Database Tier (Qdrant)
* Deploy Qdrant in **Distributed/Clustering Mode** with multiple nodes.
* Set up **sharding** to distribute points across nodes, and configure read replicas for search queries.
* Enable **Vector Quantization** (scalar quantization) inside Qdrant to reduce memory footprint by up to 4x, speeding up similarity checks by keeping the vector index entirely in RAM.

#### 4. Frontend Tier (React)
* Serve static assets via a **Content Delivery Network (CDN)** like **AWS CloudFront** to cache and deliver static files from servers physically close to the users, offloading the Nginx origin servers.

### Q7. Walk me through your database indexing strategy in PostgreSQL for the Users, Companies, and Jobs tables.
* **Answer**:
  * **Primary Keys**: PostgreSQL automatically creates indexes for Primary Keys (`id`), but we must index foreign keys explicitly to speed up `JOIN` queries.
  * **Foreign Key Indexing**: Create B-Tree indexes on `jobs.company_id` to speed up queries fetching a company's jobs.
  * **Unique Indexes**: Ensure `users.email` is indexed (automatic via `unique=True` constraints) to make user auth lookup instantaneous.
  * **Search Optimization (Trigram Index)**: For standard text search queries on Job titles or descriptions (if not using Qdrant), use a GIN (Generalized Inverted Index) with the `pg_trgm` extension in Postgres to allow fast wildcard/substring matches (`LIKE '%python%'`).
```sql
CREATE INDEX idx_jobs_company_id ON jobs(company_id);
CREATE INDEX idx_jobs_title_trgm ON jobs USING gin (title gin_trgm_ops);
```
